\chapter{Implementation Foundations}

This section establishes the foundations the methodology builds on. Section~4.1 reviews
prior custom datatype implementations, showing that the pattern \textsf{cdt:ucum} follows
is established practice rather than an isolated design. Sections~4.2 and~4.3 survey the
\ac{rdf} libraries and the \ac{ucum} backends used in the four implementations. Together
they supply the material that Step~1 of the methodology in Section~5 draws on.


\section{Implementations of Other Datatypes}

Section~2.2 established that any \ac{iri} can serve as a datatype identifier in \ac{rdf},
and that a processor recognizing a custom datatype can equip it with a full value space and
operator semantics. Three published systems show this mechanism at work in practice.

GeoSPARQL, published by the \ac{ogc}, defines a vocabulary for representing and querying
geospatial data in \ac{rdf} \cite{perry2012geosparql}. It defines two \ac{ogc} datatypes.
The datatype \textsf{geo:wktLiteral} contains the \ac{wkt} serialization of a geometry
object as defined by \ac{iso}~19125-1, while \textsf{geo:gmlLiteral} contains the \ac{gml}
serialization. A geometry is linked to its literal value through the property
\textsf{geo:asWKT} or \textsf{geo:asGML}, and each literal embeds a coordinate reference
system URI as a prefix within the lexical form. GeoSPARQL also defines a set of \ac{sparql}
extension functions for topological and geometric computation over these literals. The
standard was updated as version~1.1 in 2024 \cite{car2024geosparql}.

Hartig et al.\ present \textsf{cdt:List} and \textsf{cdt:Map}, two custom datatypes for
representing composite values as \ac{rdf} literals \cite{hartig2024datatypes}. A
\textsf{cdt:List} literal encodes an ordered sequence; a \textsf{cdt:Map} literal encodes
a set of key-value pairs. The approach introduces three \ac{sparql} extensions: a FOLD
aggregate that produces composite literals from a group of solutions, functions for accessing
and manipulating composite literals within expressions, and an UNFOLD operator that
decomposes a composite literal into one solution row per element.

Marciniak, Sowiński and Ganzha introduce a custom datatype for representing data tensors as
\ac{rdf} literals \cite{marciniak2025representing}. The motivation is the growing demand for
integrating machine learning embeddings with knowledge graphs. The lexical form encodes
tensor shape and values as a \ac{json} string. The \ac{sparql} extension defines 36
functions enabling the transformation, comparison, and concatenation of tensors, together
with four aggregates.

All three cases apply the same pattern. A custom datatype \ac{iri} defines a value space and
a lexical-to-value mapping for a domain-specific object, and \ac{sparql}-level operators or
functions are registered to expose computation over typed literals through native query
syntax.

Each of the three systems solves a different problem: spatial geometry, composite data
structures, and machine learning embeddings. What they share is the mechanism: a custom
datatype \ac{iri} with a defined value space and operator semantics exposed through
native \ac{sparql} syntax. \textsf{cdt:ucum} applies the same mechanism to physical quantities, a domain where
operator semantics must account for unit equivalence, dimensional compatibility, and
the propagation of units through arithmetic.


\section{\ac{rdf} Libraries}

\ac{rdf} libraries provide the programmatic infrastructure for working with \ac{rdf} data.
Libraries differ in which features they include and in how accessible their internals are
to extension code. Both differences matter for this thesis.

Apache Jena is an open-source Java framework for building Semantic Web and Linked Data
applications.\footnote{\url{https://jena.apache.org}} Its \ac{sparql}~1.1 engine, ARQ,
supports federated queries. Jena also supports \ac{rdfs} and \ac{owl} reasoning through
multiple configurable reasoners, \ac{shacl} validation, and a GeoSPARQL implementation.
Its current release is version~6.x.x, distributed under the Apache License~2.0.

RDFLib is a Python library for working with \ac{rdf} and the primary \ac{rdf} library in
the Python ecosystem. It provides parsers and serializers for the major \ac{rdf}
serializations and a full \ac{sparql}~1.1 implementation for queries and update statements,
together with a plugin architecture for extending stores, parsers, and serializers. Its
current release is version~7.x.x, distributed under the \ac{bsd} 3-Clause license.

rdflib.js is a JavaScript \ac{rdf} library for browsers and
Node.js.\footnote{\url{https://github.com/linkeddata/rdflib.js}} It reads and writes
RDF/XML, Turtle, and N3, and reads \ac{rdfa} and \ac{jsonld}. It functions as a
Read/Write Linked Data client, supporting \ac{webdav} and \ac{sparql}/Update for
interacting with remote data sources. rdflib.js does not ship a built-in \ac{sparql}
query engine; \ac{sparql} interaction is limited to update operations against remote
endpoints. Its current release is version~2.x.x, distributed under the MIT license.

dotNetRDF is a .NET library for parsing, managing, querying, and writing
\ac{rdf}.\footnote{\url{https://dotnetrdf.org}} It provides a complete \ac{sparql}~1.1
implementation through its built-in Leviathan in-memory query engine and supports the
major \ac{rdf} serialization formats. It also covers \ac{shacl} validation and a subset
of \ac{owl} reasoning. Its current release is version~3.5.x, distributed under the MIT
license.

\section{\ac{ucum} Libraries}

The libraries used across the four implementations fall into two categories. The first
contains libraries that parse \ac{ucum} expressions and perform unit validation and
conversion. The second contains general unit-arithmetic libraries: they maintain their own
unit vocabulary and support quantity arithmetic, but do not parse \ac{ucum} expressions.
The Python and Java implementations each pair one library from each category; the other
two use a single library.

ucumvert is a Python package that parses \ac{ucum} unit strings and converts them to Pint
quantities.\footnote{\url{https://github.com/dalito/ucumvert}} It implements the full
\ac{ucum} grammar using the Lark parser toolkit,\footnote{\url{https://github.com/lark-parser/lark}}
generating the grammar directly from the official \texttt{ucum-essence.xml} file. Pint is
the arithmetic backend: it represents quantities as numerical values paired with units,
supports arithmetic with automatic unit conversion, and raises a dimensionality error for
incommensurable quantities.\footnote{\url{https://github.com/hgrecco/pint}}

ucum-lhc is a JavaScript library providing validation and conversion services for
\ac{ucum}, developed by the Lister Hill National Center for Biomedical Communications at
the \ac{us} National Library of
Medicine.\footnote{\url{https://ucum.nlm.nih.gov/ucum-lhc}} It validates unit strings,
converts numeric values between commensurable units, and finds units commensurable with a
given expression.

Indriya is the reference implementation of the Units of Measurement \ac{api} defined by
\ac{jsr}-385,\footnote{\url{https://jcp.org/en/jsr/detail?id=385}} a Java Specification
Request defining standard interfaces for handling units and
quantities.\footnote{\url{https://github.com/unitsofmeasurement/indriya}} It covers unit
compatibility checks, conversion between units, and arithmetic on quantities. The \ac{ucum}
unit definitions are supplied separately by the uom-systems library, which plugs into
Indriya's unit framework. It is the arithmetic backend for the Apache Jena implementation
in this thesis.

Fhir.Metrics is a C\# library providing a \ac{ucum} implementation for the \ac{fhir}
standard \cite{hl7_fhir_quantity}, developed and maintained by
Firely.\footnote{\url{https://github.com/FirelyTeam/Fhir.Metrics}} It reads the
\texttt{ucum-essence.xml} data file and supports unit parsing, conversion, and quantity
comparison.
