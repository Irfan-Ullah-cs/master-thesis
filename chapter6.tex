\section{Identify Candidate Libraries}

This step looks for two things in each language: an RDF library to host the implementation and a UCUM backend to parse and compute. A short survey of each ecosystem is usually enough. RDF libraries were compared on four signals considered together: age, adoption, contributor base, and feature completeness. None is decisive alone; a library can still be the natural choice despite lacking a major feature if the community has settled on it. On the backend side the field is narrow in every language, and the key property is whether a library can parse an arbitrary UCUM expression at runtime.

For three of the four languages the survey points to one library without much contest. In Python, RDFLib has been the general-purpose RDF library for over two decades and provides a full SPARQL 1.1 engine. In JavaScript, rdflib.js is a read-write Linked Data toolkit rather than a query engine, but its long-established community role is what recommends it, the missing engine notwithstanding. In  C\#, dotNetRDF has been maintained since 2009 and ships the established in-process SPARQL 1.1 engine of the .NET world.

Java is the exception, and deliberately so. The CDT specification was accompanied by a fork of Apache Jena 3 with the datatype behaviour hardcoded into the distribution. Jena was kept here to ask whether the fork's behaviour can be recovered from the outside as a standalone extension, and to move that work to the current Jena 6.0.0. Eclipse RDF4J is the other mature Java host; Jena was chosen because the prior work sat on it.

The backends were chosen on the same mixture of capability and standing. In Java the work inherited the JSR-385 pairing from the fork: systems-ucum for the UCUM grammar and unit definitions, Indriya for conversion and arithmetic. In JavaScript, ucum-lhc, from the US National Library of Medicine, covers validation, conversion, and decomposition to SI base units. In  C\#, Fhir.Metrics, developed by Firely for the FHIR standard, parses UCUM expressions from the official definition file loaded at runtime. No single Python library covers both; ucumvert, the only native UCUM parser, is young but generated from the official UCUM definition file, and computes through Pint, the established unit-arithmetic library of the ecosystem.

What this step cannot settle is how far each host can be reached from outside. That is the question the next section takes up.


\section{Analyze Extension Points and APIs}

This step reads the internals of each host and backend. The answers vary more between the four hosts than at any other step in the section, and that variance is the finding.

Datatype registration is the one area where every host offered a usable path, though of different kinds. RDFLib's bind ties a datatype IRI to a constructor the library calls on the lexical form at parse time. Jena's TypeMapper registry calls the registered datatype's parse method on each matching literal, marking it ill-typed if the parse throws. dotNetRDF has no registration call; recognition is achieved by subclassing the graph for the cdt:ucum IRI and adding it to the engine's recognized-types list. rdflib.js offers nothing: its literal constructor performs no dispatch on the datatype IRI, so a cdt:ucum literal is stored like any other typed string.

The SPARQL operators are where the hosts diverge most, and Jena is the clearest case because its engine treats its own operators inconsistently. Equality is delegated: ARQ routes the operator through its literal layer to the registered datatype's equality method, requiring no interception. Comparison and arithmetic are not. The less-than and greater-than operators, the four arithmetic operators, the ORDER BY comparator, and the SUM and AVG accumulators all evaluate inside engine code that consults no registry and calls no method on the datatype; compiled Java offers no stable way to intervene from outside. The split is therefore imposed by the engine: one obligation reachable through a clean public method, the rest reachable only by forking.
dotNetRDF is the most accommodating: arithmetic operators go through a public registry, equality through the recognized-types list, the filter node comparer is a public settable property, and custom functions have a public factory. Only the ORDER BY comparer has no public setter; it is reachable through a single reflective access to a private field, the one non-public point in the whole extension. RDFLib exposes none of these as extension points, but Python permits replacing functions at runtime. The patching reaches further than it first appears: the operator functions live in one module, but the parser holds its own references captured when it loads, and ordering and the aggregates hold further references in separate modules; a change to the operator module alone would leave those paths running the original code while looking complete. rdflib.js has almost no SPARQL surface; its query layer handles only basic patterns and equality, comparison operators collapse to equality, ordering and aggregates are absent, there is no function registry, and the only interceptable point is a single funnel method on the native store API.

Across the three hosts with SPARQL machinery, FILTER and BIND share one evaluation path while ordering and aggregates each require separate interception, confirming the section 5 warning against assuming one hook suffices.

All four backends parse an arbitrary UCUM expression and perform dimensional arithmetic. Three signal incompatible dimensions correctly, by an exception or a failure status the extension turns into one; the fourth does not: the  C\# backend treats some distinct dimensions as the same, newton and pascal among them, so an addition that should be rejected is accepted. Section 6.3 returns to this. The numeric type is where the backends part company, and the difference sets a ceiling no outer layer can lift. The Python backend can be configured to compute in arbitrary-precision decimal once a hidden fallback to floating point in the parser's transformer is corrected. The  C\# backend computes in the platform decimal type, roughly twenty-eight significant digits, with an overflow above that range; like the Java backend, it rounds non-terminating conversion factors to that fixed precision. The JavaScript backend computes in double-precision floating point only, as defined by IEEE 754 \cite{ieee2019_754}, with no higher-precision mode, so any precision beyond a double is lost before arithmetic begins. The Java backend keeps exact precision for integer-ratio conversions but rounds non-terminating conversion factors. Coverage gaps converge on the special units of Section 3.3, which the next sections report where they surface.

The step produced, for each host, a map from every obligation to the mechanism that satisfies it or the absence of one; those maps are the input to the feasibility assessment.


\section{Assess Feasibility}

The feasibility assessment read the result from the previous section against the obligations of the specification. All four candidates returned a go; what differed was what the go required.

The backend side carries one gate: the backend must parse an arbitrary UCUM expression at runtime and perform dimensional arithmetic. All four chosen backends pass it. They part company on the numeric type they compute in, and the feasibility step accepts that ceiling because no outer layer can raise it, from arbitrary-precision decimal in Python to double precision in JavaScript.

For Jena, the go required accepting that comparison and arithmetic cannot be reached from outside the library; compiled Java offers no way in without a fork. Registration, equality, and the sameDimension function are met through public APIs, so Jena still parses CDT literals, recognizes them, and answers queries that rest on parsing and equality.

For rdflib.js, the go rests on the same reasoning at a wider scale. The library has no registration mechanism and a query layer that collapses comparison operators to equality, so most operator obligations cannot be reached inside a query at all. It is still the library its community reaches for; the obligations it can meet are reframed onto the native store API, and the rest are delivered through an explicit operations layer.
For dotNetRDF, every obligation is reachable on the query surface through public registries and one reflective access for the ordering comparer. One backend limitation is recorded: the Fhir.Metrics dimensional check compares unit axes without their exponents, treating some distinct dimensions as the same, newton and pascal among them, so an addition the specification requires to fail is accepted. This is a defect in the backend, not the extension, and the go is taken with it documented.

For RDFLib, the go is the most complete of the four. Every obligation is met: three through public APIs, the other two by replacing operator functions at runtime, with those replacements reversible and producing results the specification accepts. The backend computes in arbitrary-precision decimal once a single fallback in the parser is corrected.

All four are a go. dotNetRDF and RDFLib reached it with every obligation on the query surface; Jena and rdflib.js reached it by moving some off. The rule against forking the host is what separates the two pairs. Underneath all four sits the same idea: the obligations shape the decision, but a host is chosen because it is useful and used, and a partial fit on a library a community trusts is worth more than a full fit on one nobody runs.


\section{Implementation}

The four implementations followed the same fixed order as Section 5 prescribes: representation, registration, SPARQL hooks, custom function, and edge cases.

The representation class in each implementation wraps the backend and exposes parsing, equality, comparison, arithmetic, conversion, and the dimension test, realising the value space of Section 3.4; all backend access is confined to it.

Registration tied cdt:ucum and cdt:ucumunit to the representation through the mechanism named in the feasibility record, after which the host parses a CDT literal into a typed object rather than an opaque string. Where the library attaches no behaviour to a datatype IRI, recognition lives inside the extension and is applied at every point where literals enter it; rdflib.js works this way.

The hooks follow the same pattern in every implementation: check operands, evaluate CDT pairs through the representation, pass everything else through unchanged, and catch ill-typed literals before any value operation. What most separates the four is the number of interception points needed. In Jena and dotNetRDF, registration alone delivered SPARQL equality because the engine routes equality through the registered datatype.

The custom function cdt:sameDimension extracts two quantities, asks the backend whether their dimensions match, and returns a boolean.

The same five cases arose in every implementation: unparseable units, with the special units of Section 3.3 foremost; cross-dimension comparison in ORDER BY, which must never abort the query; unit stripping in SUM and AVG accumulators; precision loss at numeric extremes; and the dimensionless case, which more than one backend required a dedicated branch to handle. Each was handled explicitly and the chosen behaviour recorded, because several return in the next step as expected failures.

\section{Validation}

Before any library was tested, a single reference set of behavioral requirements was drawn up, setting out what a complete implementation should be tested for, independent of any one language. Each of the four suites is read against that same reference, and every requirement receives a status: covered, covered but behaving differently in a documented way, not applicable to the language, or absent. 

The four suites differ in size, but size is not a conformance score. RDFLib has the most at 253 tests, none failing. dotNetRDF has 251, of which thirty-eight are left failing; the discussion below records which cases take which form. Jena has 163 and rdflib.js 157. What the totals hide, the distribution shows, because the shape of each suite tracks where its operators live. In RDFLib the operators run through the engine, and the query file alone holds sixty-nine tests. In rdflib.js the same obligations are reframed onto the store API, and the query file holds nine, with the weight moved to parsing and arithmetic on the quantity class directly. Jena has no query-level test file at all, and its largest class, sixty-seven tests, exercises the Java operations API that comparison and arithmetic were reframed onto.

Each obligation is tested at the level at which the feasibility record claims it. The Python implementation tests at both levels: the quantity class on its own, and the operators, ordering, and aggregates through queries the engine runs. The Java implementation exercises equality through the datatype interface but submits no query, so query-level equality rests on Jena's documented routing rather than on a test of the extension. Arithmetic was tested across its full type coverage; two of the four implementations were incomplete at the same point.

The edge cases from implementation were then tested: a query over data containing an ill-typed literal must complete with the affected solutions excluded; ordering over mixed dimensions must complete; aggregates must preserve units where claimed; a literal written out and parsed back must keep its value; and activating twice must change nothing.

Expected failures mark the obligations the feasibility record left as not achievable, with the cause attached to each. Two families recur across all four: the special units of Section 3.3 and precision loss in bounded numeric types. The form differs by backend. Where lower precision is the accepted behaviour, the boundary becomes a passing test that documents the loss: rdflib.js records a value beyond double range collapsing to infinity and the sixteenth-digit case collapsing to one float. dotNetRDF records all limitations as failing tests: its decimal backend overflows above roughly twenty-eight significant digits and rounds non-terminating conversion factors; the latter is why a cross-unit equality such as 3.6 km/h against 1 m/s does not hold. The Celsius offset conversion and the Fhir.Metrics dimensional defect that reports newton and pascal as one dimension are also left as failing tests with the cause attached. In all, the dotNetRDF suite leaves thirty-eight tests failing.



\section{Findings Across the Cases}

The clearest finding is that the same obligation landed differently in each host for one reason: how far an extension could reach from outside. The host's architecture determines this, whether operators are exposed through public points or evaluated in code an extension cannot enter, and so does the host language. A dynamic language permits replacing functions at runtime, which is how the Python operators were reached at twenty-five points without altering the library; a compiled language offers no such entry. 

The rule against forking sorts the four into three positions: RDFLib and dotNetRDF with every obligation on the SPARQL surface; Jena with comparison and arithmetic reframed onto an explicit Java API while equality stays on the surface; and rdflib.js with no complete SPARQL engine. All four stay worth implementing because they are the libraries their communities use.
